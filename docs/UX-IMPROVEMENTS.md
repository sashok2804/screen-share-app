# UX/UI Improvements Roadmap

> Аудит текущего веб-клиента `screen-share-app` + исследование аналогов (Discord,
> Zoom, Google Meet, Microsoft Teams, Twitch/YouTube Live) + prioritized roadmap
> для сценария **«смотреть кино/геймить вместе с друзьями»**.
>
> Дата аудита: 2026-07-21. Стек: React 19 + TypeScript + Tailwind v4. Чтение
> компонентов: `Lobby`, `Room`, `VideoStage`, `ParticipantList`,
> `StreamControls`, `SubscribeButton`, `AudioMeter`, `SourcePicker`.
> Протокол сигналинга: `server/src/protocol.ts`.

---

## Audit Summary

Текущее состояние — это **технически зрелый MVP**, но **продуктово сырой**
инструмент. Из коробки приложение решает ровно одну задачу: «показать свой экран
в 1440p60 через WebRTC без сервера-ретранслятора». UX при этом рассчитан на
**автора демо**, а не на конечного пользователя, который хочет посмотреть кино
с друзьями.

**Сильные стороны:**

- Корректная mesh P2P-архитектура с perfect-negotiation, polite/impolite glare
  control. Это редкая зрелость для pet-project.
- Гибкая модель подписок: зритель сам решает, смотреть ли стрим — это экономит
  трафик и явно отражено в UI («Подписчиков: N», «трафик не идёт»).
- 5 пресетов качества с honest effective-resolution индикатором (если монитор
  1080p — `ultra` вернёт 1080p, и UI это показывает).
- AEC-предупреждение в UI: жёлтая плашка честно говорит «друг может слышать эхо
  своего голоса → ставь Electron-клиент». Это зрелая прозрачность.
- VU-метры у каждого участника (4 зоны цвета: silent/talking/loud/clipping) —
  маленькая, но очень полезная фича, которой нет даже в Discord.
- Поддержка Electron-only SourcePicker с превью окон/экранов — намного лучше
  нативного диалога Chrome.

**Слабые стороны (по убыванию критичности для watch-party сценария):**

1. **Нет чата.** Смотреть кино с друзьями без возможности написать «лол» или
   « перемотаай» — бессмысленно. Это **P0**.
2. **Нет sharing-ссылки.** Чтобы позвать друга, надо продиктовать `roomId`
   голосом или через сторонний мессенджер. При этом URL уже содержит `?room=`
   (`App.tsx:21-26`) — кнопки «Copy invite» просто нет.
3. **Нет индикации latency/качества стрима.** У зрителя никак не видно,
   идёт ли 60 fps или деградировало до 15. `chrome://webrtc-internals` — это
   инструмент разработчика, не пользователя.
4. **«Плашка про стрим» показывается только до первой подписки.** После того
   как зритель нажал «Смотреть», в UI нет отдельного подтверждения «хост
   стримит, вы смотрите» — только бейдж `● LIVE` на видео.
5. **У хоста нет способа попросить тишины.** Mute-кнопка есть только у себя;
   нельзя замьютить шумного участника («тап-тап по микрофону»).
6. **Звук «по умолчанию непонятно как».** При нажатии «Начать стрим»
   `getDisplayMedia` открывает нативный диалог Chrome, где пользователь должен
   **сам** поставить галочку «Share audio» и выбрать «Entire screen». UI apps
   не подсказывает этого заранее — типичный пользователь выберет «Окно плеера»
   и получит немое кино.
7. **Подписка — это onclick-кнопка в отдельной плашке** под видео. У Twitch/
   Discord subscribe — это один клик в точке внимания (на самом видео).
8. **У зрителя нет управления громкостью хоста отдельно от голоса.** Если
   взрывной саундтрек фильма в 3 раза громче голосов друзей — зритель ничего
   не может сделать, кроме как ползунка системной громкости вкладки.
9. **Список участников показывает VU, но не показывает кто смотрит, а кто
   просто сидит с голосом.** Бейдж `👁` невзрачный, и не даёт ответа «сколько
   из 5 человек сейчас реально смотрят».
10. **Нет reactions.** Discord/Meet/Twitch — все имеют emoji-reactions. Для
    watch-party это **primary channel feedback** (аплодисменты в конце сцены,
    «🔥» на ярком моменте).
11. **Lobby не запоминает имя** и не валидирует занятость roomId. Если зайти
    в чужой roomId — попадёшь к чужим, без предупреждения.
12. **Нет picture-in-picture** у зрителя. Если зритель хочет параллельно
    открыть чат в Discord или браузере — видео теряется.

Подробности и приоритеты — ниже.

### Что уникально у нас (не надо терять)

- **1440p60 + AV1/VP9** — у Discord Go Live максимум 1080p60.
- **Честная mesh P2P** — zero server media cost. Twitch/YouTube не могут этого.
- **AEC-warning UI** — ни у кого нет явного предупреждения про echo loop.
- **Electron + FFmpeg WASAPI** — работает там, где браузер принципиально не
  может (захват отдельного приложения со звуком).
- **Subscriber-driven traffic** — зритель решает, дать ему трафик или нет.

## Proposed Features (Prioritized)

Приоритеты считались для главного сценария: **3–6 друзей хотят вместе
посмотреть фильм/стрим YouTube/поиграть в кооператив**. Не для «корпоративная
презентация на 50 человек» (это вне текущей mesh-модели).

### P0 — Must Have (blocking core use case)

Без этих фич смотреть кино с друзьями **физически неудобно**.

#### P0.1 — Текстовый чат в правом сайдбаре

**Сейчас:** отсутствует.
**Надо:** правый сайдбар 320px, история последних 50 сообщений, autoscroll,
`Shift+Enter` = перенос, `Enter` = отправить, badge с непрочитанными при
collapsed state. Системные сообщения («Alice начала стрим», «Bob подписался»)
в отдельном стиле (italic, slate-500).

**Wire-протокол:**
```ts
// Client → Server
{ type: 'chat', payload: { text: string } }   // text.length ≤ 500, trim
// Server → Client (broadcast)
{ type: 'chat', from: '<participantId>', payload: { text: string, ts: number } }
```
Сервер хранит только последние 50 (ring buffer), отдаёт при join.
Throttle: 3 msg/sec на клиента, иначе drop.

**UI:** В `Room.tsx` — третья колонка `<ChatPanel>` после `ParticipantList`.
На mobile (<768px) — переключаемая вкладка «Участники / Чат».

#### P0.2 — Кнопка «Пригласить по ссылке» + copy-to-clipboard

**Сейчас:** URL уже содержит `?room=` (`App.tsx:21-26`), но пользователь об
этом не знает.
**Надо:** в header рядом с `room: demo` — иконка 📋 (copy). При клике:
`navigator.clipboard.writeText(window.location.href)` + toast «Ссылка
скопирована» (2 сек). Дополнительно — кнопка «Поделиться» открывает native
`navigator.share()` на mobile (Web Share API).

В Electron — добавить генерацию deep-link `screen-share://room/<roomId>` с
помеченной «Открыть в приложении».

**Wire-протокол:** без новых сообщений.

#### P0.3 — Индикатор качества соединения у зрителя

**Сейчас:** ничего. Если стрим лагает — пользователь не понимает, это у него
интернет отвалился или у хоста.
**Надо:** `useConnectionQuality()` hook на основе `RTCStatsReport` (_poll каждые
2 сек из `RTCPeerConnection.getStats()`). Считаем:

| Метрика | Источник в stats | Порог |
|---|---|---|
| RTT | `remote-inbound-rtp.roundTripTime` | < 100ms=green, < 300ms=yellow, else red |
| Packet loss | `remote-inbound-rtp.packetsLost / packetsReceived` | < 1%=green, < 5%=yellow, else red |
| FPS (декодировано) | `inbound-rtp.framesPerSecond` | сравнить с target пресета |
| Bitrate | `(bytesReceived - prevBytesReceived) / dt` | сравнить с preset.maxBitrate |

**UI:** Badge `● 1080p60 · 42ms · 16.2 Mbps` в левом-верхнем углу `VideoStage`
(заменяет текущий `resolutionLabel`). При деградации — отдельная «⚠ Низкое
качество» pill в правом-верхнем, с tooltip про причину.

#### P0.4 — Громкость хоста и громкость голоса — раздельно

**Сейчас:** у зрителя один `<audio>` элемент на всех (`voice.remoteAudioRef`).
Нельзя сделать кино потише, не заглушив друзей.
**Надо:** две регулируемые шины:

1. **Voice bus** — все микрофоны участников (включая хоста).
2. **Stream audio bus** — системный звук хоста ( музыка/диалоги фильма).

У каждого `<audio>` элемента — свой `GainNode` через `AudioContext`. UI:
вертикальный мини-миксер (только у зрителя) — два слайдера 0–100% + master mute.
Разместить в `StreamControls.tsx` снизу (заменить текущий grid с «Активный
пресет»).

Это требует рефактора в `useVoice.ts` + `useScreenShare.ts`: сейчас remote
audio одной дорожкой идёт в `<audio>`. Надо развести: voice tracks →
`voiceGainNode` → destination, stream audio track → `streamGainNode` →
destination.

#### P0.5 — Reactions (emoji overlay)

**Сейчас:** ничего.
**Надо:** кнопка-ряд внизу видео-стейджа с 6 эмодзи: 👍 ❤️ 😂 🎉 🔥 👏.
Клик → летит на сервер → рассылается всем. Зритель видит floating emoji
которое всплывает снизу вверх и исчезает за 3 сек (random X position, 60–100px
размер). При массовых одинаковых реакциях (>3 за 2 сек от разных людей) —
«combo» анимация (крупнее).

**Wire-протокол:**
```ts
{ type: 'reaction', payload: { emoji: '👍' } }   // client → server
{ type: 'reaction', from: '<id>', payload: { emoji: '👍', ts: number } }  // broadcast
```
Throttle: 2 reactions/sec от одного клиента.

**UI:** новый компонент `<ReactionBar>` (overlay на `VideoStage`) +
`<ReactionOverlay>` для floating emoji. CSS animation, никаких тяжёлых
библиотек.

### P1 — Should Have (significant UX improvement)

#### P1.1 — Pre-flight чеклист перед «Начать стрим»

**Проблема:** пользователь жмёт Start → видит нативный диалог Chrome → выбирает
«Окно плеера» → получает немое кино → пишет в чат «у вас есть звук?».
**Решение:** модальное окно **перед** вызовом `getDisplayMedia`:

```
┌────────────────────────────────────────────┐
│  Подготовьтесь к стриму                     │
├────────────────────────────────────────────┤
│  ✓ Выберите источник:                       │
│     ○ Весь экран (рекомендуется для кино)   │
│     ○ Окно приложения                       │
│     ○ Вкладка браузера                      │
│                                             │
│  ⚠ Звук:                                    │
│     [ ] Включить системный звук             │
│     ⚠ В Chrome звук доступен только         │
│       для «Весь экран» или «Вкладка»        │
│     💡 Для захвата отдельного приложения    │
│       со звуком — установите desktop-клиент │
│                                             │
│  🎤 Микрофон: включен (значит друзья        │
│     будут слышать ваши комментарии)         │
│                                             │
│  Качество: 1440p60 (ultra)                  │
│  [ Отмена ]   [ Начать демонстрацию → ]     │
└────────────────────────────────────────────┘
```

Этот модал появляется **до** нативного диалога Chrome и не заменяет его
(браузер всё равно покажет свой), но подготавливает пользователя. В Electron —
`SourcePicker` уже решает это, можно расширить.

#### P1.2 — Picture-in-Picture для зрителя

**Сейчас:** видео занимает только `VideoStage` внутри нашей вкладки. Если
зритель переключается на другую вкладку — видео ставится на паузу (Chrome
polite autoplay policy).
**Надо:** кнопка «⧉ PiP» в правом-верхнем углу `VideoStage`. Вызывает
`video.requestPictureInPicture()` (Chrome/Edge/Firefox поддерживают). Зритель
видит плавающее окно поверх всех приложений, может параллельно работать.

В Safari iOS PiP работает тоже. Документация: MDN
`Picture-in-Picture_API`. Спецификация: W3C Draft.

#### P1.3 — «Кто смотрит» — отдельная секция в ParticipantList

**Сейчас:** у подписчика бейдж `👁`, но он теряется среди других бейджей.
**Надо:** разбить `ParticipantList` на две подсекции:

```
┌──────────────────────────────┐
│ В комнате · 5                │
├──────────────────────────────┤
│ Смотрят стрим · 3            │
│  • Alice (host) ●━━━━━━━━    │
│  • Bob           ●━━░░░░░    │
│  • Carol         ●━━━━━━━    │
├──────────────────────────────┤
│ Только голос · 2             │
│  • Dave          ●━━━━━      │
│  • Eve (muted)   ░░░░░░░     │
└──────────────────────────────┘
```

Реализация — простая группировка массива `participants` по
`subscribers.includes(p.id)`. В `useRoom.ts` уже есть `subscribers: string[]`.

#### P1.4 — Hand raise (поднять руку)

**Сейчас:** зритель может только писать в чат или говорить. В шумной комнате
или при mute — никак не привлечь внимание хоста.
**Надо:** кнопка ✋ в `Room.tsx` header (между mic toggle и Leave). При
активации:

- В `ParticipantList` у этого юзера появляется большая иконка ✋ (amber-400,
  pulse).
- Хосту — toast «Alice подняла руку» (3 сек).
- Повторный клик — опускает руку.

**Wire-протокол:**
```ts
{ type: 'hand-raise', payload: { raised: boolean } }   // client → server
{ type: 'hand-state', from: '<id>', payload: { raised: boolean } }   // broadcast
```
Сервер хранит `raisedHands: Set<participantId>` в room state, рассылает в
`peer-list` (расширить payload).

#### P1.5 — Хост может mute участников

**Сейчас:** `voice.toggleMic` мутирует только себя.
**Надо:** в `ParticipantList` у хоста (или owner) рядом с каждым участником —
иконка mute. Клик → сервер шлёт `mute-request` этому participant → его клиент
мьютит свой mic (с системным уведомлением «Хост замьютил вас»).

**Wire-протокол:**
```ts
{ type: 'mute-request', to: '<participantId>' }   // host → server
{ type: 'mute-requested', payload: {} }           // server → target
```
Только host/owner может слать `mute-request`. Сервер валидирует роль.

UI — только для хоста/owner. У обычного зритителя этой кнопки нет.

#### P1.6 — Auto-subscribe на стрим + persistent subscribe

**Сейчас:** когда хост начинает стрим, у каждого зрителя отдельно появляется
плашка «Хост запустил стрим — Смотреть». Надо нажать.
**Надо:** два режима:

1. **Auto-subscribe** — пользователь один раз отметил в настройках «Always
   watch when host streams». Дальше — автоматически подписывается.
2. **Persistent** — если зритель уже смотрел стрим этого хоста в этой
   комнате, потом отпался, а хост перезапустил стрим — автоматически
   переподключается.

Локально храним `localStorage['ss.autoSubscribe'] = true|false`. Дефолт —
false (уважаем трафик), но включается одним тумблером в `Lobby`.

При auto-subscribe — короткий toast «Подключаюсь к стриму…» 1 сек, чтобы
пользователь понимал, что произошло.

#### P1.7 — Stream Health Indicator у хоста

**Сейчас:** хост видит «Подписчиков: 3» и битрейт «16 Mbps × 3». Не видно
качества стрима у зрителей.
**Надо:** под этими цифрами — список подписчиков с их latency (round-trip):

```
Подписчиков: 3 · 48 Mbps uplink
  • Alice   42ms  ✓ 1080p60
  • Bob    187ms  ⚠ деградация до 720p30
  • Carol   58ms  ✓ 1080p60
```

Данные: у хоста уже есть `RTCPeerConnection` на каждого зрителя, можно polled
`getStats()` и сравнить `outbound-rtp` с `inbound-rtp` у зрителя (зритель
репортит обратно через сигналинг — `stats-report` message каждые 5 сек).

### P2 — Nice to Have (polish)

#### P2.1 — Запомнить имя и roomId в Lobby

#### P2.2 — Soundboard (короткие звуки)

6 кнопок в `<ReactionBar>` рядом с emoji: аплодисменты, смех, грусть, барабаны,

| `M` | toggle mic |
| `V` | toggle video (если хост) / toggle subscribe (если зритель) |

Перед входом в комнату — показать свой IP region, RTT до STUN-сервера, и

Сейчас только dark. Light тема не приоритетна для watch-party (всё равно
кино смотрят в темноте), но полезна для презентаций днем.

#### P2.6 — Camera tile (для зрителей)

Опционально — каждый участник может включить веб-камеру (отдельным transceiver

streamer cam). Это уже близко к Discord voice channels.

#### P2.7 — Sync playback indicator

Для watch-party YouTube/Netflix — детектить, что хост на паузе, и показывать
зрителю «⏸ Host приостановил воспроизведение». Реализуется через
`useScreenShare.ts` hook: периодически (1/sec) слать metadata frame
через `RTCDataChannel` с `document.querySelector('video')?.currentTime`.
Сложно для Netflix (DRM), но для YouTube работает.

#### P2.8 — Поддержка глубоких ссылок с timestamp

`?room=demo&t=60` — зритель заходит на 60-й секунду. Полезно для VOD (если
будет запись).

#### P2.9 — Notifications (OS-level)

При упоминании имени в чате или при «hand raise» — системная нотификация
`Notification.requestPermission()` + title + body. В Electron — нативные
тосты.

#### P2.10 — Pre-stream preview для хоста

Превью того, что будет транслироваться (как в OBS), перед Go-Live. Для
браузера — `getDisplayMedia` сразу стримит, но можно показать preview 3 сек
перед `notifyStreamStart()`.

#### P2.11 — Recording (VOD)

Из README — это уже в Phase 2 roadmap. `MediaRecorder` на стороне зрителя (или
сервера в SFU-режиме) → chunked upload → каталог прошлых стримов в новой
секции `Lobby`.

#### P2.12 — Emoji-реакции прямо в чате

Парсинг `:smile:`, `:fire:` в сообщениях чата → замена на unicode. Библиотека
не нужна, словарь на 50 эмодзи достаточен. Учёт reactions count у сообщения
(твиттер-стайл «❤ 3» под сообщением).

## Visual Design Direction

### Палитра

Текущая схема `slate-950 / slate-900 / indigo-600 / emerald-500 / rose-600` —
**хорошая база**, не надо переделывать. Дополнения:

- **Accent gold** (`amber-400`) — для host/owner отличий (уже используется в
  `roleBadge('owner')`, консистентно расширить).
- **Live red** (`rose-500`) с pulse-анимацией для `● LIVE` индикатора
  (сейчас `rose-600` статичный — добавить `animate-pulse`).
- **Quality colors** (для connection badge):
  - green: `emerald-400` (good)
  - yellow: `amber-400` (warning)
  - red: `rose-500` (bad)

### Типографика

- Основной шрифт: системный stack (`-apple-system, 'Segoe UI', system-ui, sans-serif`).
  Сейчас в Tailwind v4 это дефолт — ок.
- Mono для технических данных (latency, bitrate, resolution): `font-mono` —
  уже применяется в `Room.tsx:123`. Расширить на все бейджи качества.
- Размеры: текущая шкала (`text-xs` 12 / `text-sm` 14 / `text-base` 16) —
  хорошая. Не увеличивать.

### Spacing & Layout

- **Header:** `py-3 px-6` — ok.
- **Main grid:** сейчас `md:flex-row gap-4 p-4`. После добавления чата —
  перейти на CSS Grid:
  ```
  grid-template-columns: 1fr 288px 320px;  /* stage | participants | chat */
  ```
  На <1024px — `1fr`,Participants и Chat — табы.
- **Border radius:** консистентно `rounded-xl` (12px) для карточек,
  `rounded-lg` (8px) для кнопок, `rounded-md` (6px) для бейджей. Сейчас
  смешано (где-то `rounded-2xl`, где-то `rounded-lg`) — унифицировать.

### Иконки

- **Сейчас:** SVG inline (один в `VideoStage`, один в `SourcePicker`). Это
  плохо масштабируется.
- **Рекомендация:** подключить [`lucide-react`](https://lucide.dev)
  (~3 KB на иконку, tree-shakeable). Совпадает по визуалу с текущим стилем
  (1.5 stroke, 24 viewBox). Заменить:
  - `🎤` → `Mic` / `MicOff`
  - `🔇` → `VolumeX`
  - `👁` → `Eye`
  - `📋` → `Link` (для invite)
  - `✋` → `Hand`
  - `⧉` → `PictureInPicture2`

### Анимации

- Все анимации через Tailwind (`animate-pulse`, `transition`, `duration-75`).
  Без внешних библиотек.
- Для reactions overlay — CSS keyframes:
  ```css
  @keyframes floatUp {
    0%   { transform: translateY(0)    scale(0.5); opacity: 0; }
    20%  { transform: translateY(-20px) scale(1);   opacity: 1; }
    100% { transform: translateY(-200px) scale(1);  opacity: 0; }
  }
  ```

## Конкретные ответы на вопросы

**1. Как лучше показать что хост стримит, а зритель может подключиться?**

Сейчас: плашка под видео. Проблема: при длинном стриме плашка висит indefinitely, привлекает внимание дольше нужного.

Лучше:
- **При старте стрима** — полноэкранный тост на `VideoStage` (занимает весь
  idle-place видео-плейсхолдера): «🎬 Alice запустила стрим» + большая кнопка
  «▶ Смотреть». Через 10 сек или при клике — коллапсится в маленький badge в
  углу `VideoStage`.
- **После подписки** — badge `● LIVE` на видео + индикатор качества.
- **Приick на idle stage** (там где написано «Стрим ещё не запущен»)
  должна быть кнопка-CTA, а не пассивный текст.

**2. Как решить проблему «звук приложения» — может UI должен подсказывать выбор источника?**

Да. См. P1.1 — pre-flight модал. Главная идея: **показать этот модал ДО
нативного диалога Chrome**, чтобы пользователь уже знал, что ему выбирать.
Объяснить 3 варианта (Весь экран / Окно / Вкладка) и явно отметить,
у какого есть звук.

Дополнительно:
- В `Lobby` — секция «Частые проблемы» со ссылкой на Electron-клиент.
- В `StreamControls` после старта — если `effective.hasAudio === false`,
  крупная плашка «⚠ Звук не захватывается. Скорее всего, вы выбрали
  отдельное окно. Перезапустите с выбором "Весь экран"». Сейчас есть
  бейдж «🔇 без звука», но он太小 и невзрачный.

**3. Нужен ли текстовый чат в комнате? Как его разместить?**

Да, **P0.1**. Разместить: правый сайдбар 320px (сейчас `ParticipantList` 288px
в этой роли — добавить третью колонку). На <1024px — табы
«Участники | Чат». Высота — на всю высоту main area, scroll внутри. История
50 сообщений на сервере (ring buffer), при join — отдаётся.

**4. Нужен ли список «кто смотрит» / «кто在线»?**

Да, **P1.3**. Данные уже есть (`subscribers: string[]` в `useRoom`). Надо
визуально разбить `ParticipantList` на «Смотрят стрим» / «Только голос».
Дополнительно: в `StreamControls` у хоста — показывать каждого подписчика с
его latency (P1.7).

**5. Как показать задержку (latency) и качество соединения?**

`useConnectionQuality()` hook на `RTCPeerConnection.getStats()` каждые 2 сек.
Считать RTT, packet loss, FPS декодирования, bitrate. UI: badge на
`VideoStage` `● 1080p60 · 42ms · 16.2 Mbps` (P0.3). При деградации —
**второй** badge `⚠ Низкое качество` с кнопкой «Сменить пресет» (для SVC —
автоматический fallback; в mesh — через просьбу хосту).

**6. Как добавить реакции (эмодзи, аплодисменты)?**

P0.5. Compoнент `<ReactionBar>` (overlay внизу `VideoStage`) + `<ReactionOverlay>`
(для floating emoji). Wire message: `{type:'reaction', payload:{emoji}}` →
broadcast. Анимация CSS keyframes `floatUp`. Throttle: 2/sec/user.
Soundboard (P2.2) — рядом, кнопки с короткими звуками (WebAudio API),
throttle 1/5sec.

**7. Нужен ли PiP (picture-in-picture) для зрителя?**

Да, **P1.2**. Web standard `video.requestPictureInPicture()` — поддерживается
в Chrome/Edge/Firefox/Safari iOS. Кнопка `⧉` в правом-верхнем `VideoStage`.
Критично для сценария «зритель хочет параллельно писать в Discord / гуглить
по сюжету» — без PiP он потеряет видео при переключении вкладки.

**8. Как лучше сделать управление громкостью каждого участника?**

Два уровня:
- **Individual volume** — у каждого участника в `ParticipantList` (hover на
  нём) мини-слайдер 0–100%. Реализация: `MediaElementSourceNode` →
  `GainNode` per remote track. В `useVoice.ts` сейчас все remote tracks
 attachятся к одному `<audio>` — нужно разводить по отдельным `GainNode`.
- **Bus volume** — voice bus / stream audio bus (P0.4). Это **важнее**
  individual: у большинства зрителей проблема «фильм громче голоса», а не
  «Боб громче Алисы».

Приоритет — сначала P0.4 (bus mixer), потом per-participant.

**9. Нужна ли возможность пригласить по ссылке (share link)?**

Да, **P0.2**. URL уже содержит `?room=` (см. `App.tsx:21-26`) — нужно просто
предоставить UI. Кнопка `📋 Invite` в header → dropdown с тремя опциями:
- Copy link (`window.location.href`)
- Copy deep-link (`screen-share://room/<id>`) — для Electron users
- Web Share (`navigator.share()`) — на mobile

Дополнительно — QR-код в dropdown для входа с телефона (camera scan).
Библиотека `qrcode` ~20 KB gzip.

**10. Как показать управление хостом (передача, mute других)?**

- **Передача хоста** — уже есть (`onTransferHost` → кнопка «→ хост» у owner).
  Но спрятана в `ParticipantList`. Сделать отдельную dropdown-меню
  у каждого участника (правый клик или ⋯): «Сделать хостом», «Mute»,
  «Понизить руку», «Передать owner».
- **Mute других** — P1.5. Wire: `{type:'mute-request', to}`. Только host/
  owner. На клиенте получателя — форсить `voice.toggleMic` + toast «Хост
  замьютил вас». Запретить unmute (только сам юзер может раздать звук
  обратно) — стандартная этика в Zoom/Meet.
- **«Mute All»** — кнопка внизу `ParticipantList` для host. Мьютит всех
  кроме хоста одним кликом, с подтверждением.

## Финальная нота

Этот документ — **actionable roadmap**, не Vision. Каждая фича описана с
wire-протоколом, UI-размещением, зависимостями. Любой инженер в команде
может взять P0.x и реализовать без дополнительных вопросов.

**Главный месседж:** текущий product — это excellent tech demo, но чтобы
стать продуктом «кино с друзьями», не хватает **трёх ключевых вещей**:
текстового чата, reactions и mixer'а громкости. Это **P0**, всё остальное
— polish.
